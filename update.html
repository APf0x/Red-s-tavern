GitHub Actions Tutorial

Introduction

GitHub Actions is a powerful tool that allows you to automate workflows directly in your GitHub repository. You can use it to build, test, and deploy your projects.

Getting Started

To get started with GitHub Actions, you need to create a workflow file named .github/workflows/main.yml in your repository.

Creating a Simple Workflow

Let's create a simple workflow that runs on every push to the main branch:


            name: Simple Workflow
                    
            on:
              push:
                branches:
                  - main
                    
            jobs:
              build:
                runs-on: ubuntu-latest
                    
                steps:
                  - name: Checkout repository
                    uses: actions/checkout@v2
                    
                  - name: Set up Node.js
                    uses: actions/setup-node@v4
                    with:
                      node-version: '14'
                    
                  - name: Install dependencies
                    run: npm install
                    
                  - name: Build
                    run: npm run build

Understanding the Workflow

GitHub Actions workflows are defined in a YAML file, typically located in the .github/workflows directory of your repository. Let's explore the key components of the provided workflow:

  • Trigger: The trigger specifies when the workflow should run. In this case, the workflow is configured to execute on every push event to the main branch. This ensures that changes made to the main branch trigger an automated build and test process.
  • Job: A job is a unit of work that can run on the same machine or in parallel on different machines. The defined job is named "build" and is set to run on the latest version of the Ubuntu environment. This choice of environment ensures consistency and reproducibility across different workflow runs.
  • Steps: Steps are the individual tasks that make up a job. Each step is a command or an action that GitHub Actions will run. In this workflow:
    • The first step, "Checkout repository," uses the actions/checkout action to fetch the latest version of the code from the repository. This is a crucial step as it provides the workflow with the codebase to operate on.
    • The second step, "Set up Node.js," utilizes the actions/setup-node action to set up the Node.js environment. This is essential for JavaScript-based projects.
    • The third step, "Install dependencies," runs the command npm install to install project dependencies. This assumes the presence of a package.json file with defined dependencies.
    • The fourth step, "Build," executes the build script using npm run build. The specifics of the build script depend on your project and may involve tasks like compiling code, running tests, or generating artifacts.

Viewing Workflow Runs

The "Actions" tab in your GitHub repository is the central hub for monitoring workflow runs. Once the workflow is triggered, you can navigate to this tab to view a detailed log of each run. The log provides information on every step, including the commands executed and their outcomes. It's a valuable tool for diagnosing issues, verifying successful builds, and understanding the overall workflow performance.

Expanding the Workflow

GitHub Actions offers extensive customization possibilities through the use of actions. Actions are reusable units of code that can be combined to create powerful workflows. The GitHub Actions Marketplace is a repository of community-contributed actions that cover a wide range of tasks. You can explore this marketplace to find actions that suit your project's needs, whether it's deploying to a cloud service, sending notifications, or automating other development tasks.

Additionally, you can expand your workflow by incorporating features such as conditional execution, matrix builds for testing across multiple environments, and even integrating with third-party services. The flexibility of GitHub Actions empowers you to tailor your automation to the specific requirements of your project.
